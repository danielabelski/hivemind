# PR #97 follow-ups (uncovered by E2E test 2026-05-07)

PR #97 (`feat/centralize-autoupdate`) is centralization-complete for the
4 agents whose session-start hook spawns the helper directly: claude-code,
codex, hermes, pi. Cursor and OpenClaw passed the source-level checks but
have IDE/sandbox-specific limitations the PR doesn't fully resolve. These
are tracked here as discrete follow-ups.

## Cursor IDE — detached spawn from inside hook fails to complete

### Symptom

When Cursor's `sessionStart` hook fires inside the IDE (real agent chat
started in a Cursor workspace), the autoUpdate helper's debug log line
"agent=cursor entered" + cache-file mtime confirm the trigger fires. But
the detached `hivemind update` it spawns never completes the upgrade:

  - global hivemind binary stays at 0.6.99 after the IDE session
  - `~/.deeplake/.autoupdate-last-check` IS touched (helper dispatched
    something) — so `autoUpdate` ran to completion in-helper
  - the spawned `hivemind update` either silently failed or its npm
    install -g step had no effect

### Confirmation that the upgrade chain itself is fine

`hivemind update` invoked from a normal terminal (with the same
verdaccio routing in `~/.npmrc`) upgrades global 0.6.99 → 0.7.99 in
~10 seconds. So the chain is correct; only the Cursor-IDE-spawned
invocation fails.

### Likely root causes

1. **Snap sandbox env**. Cursor at `/snap/bin/cursor` runs with restricted
   filesystem and child-process environment. The detached child may not
   have access to:
     - `~/.npmrc` (which carries the `@deeplake:registry=...` routing for
       a hypothetical user-on-verdaccio + the auth token for their real
       npm publishes)
     - `/usr/local/bin/npm` or the nvm-managed `npm` (PATH inheritance
       may be empty or restricted)
2. **stdio: "ignore"** in our `defaultSpawn` discards stdout/stderr, so
   any error from `npm install -g` is invisible. We get no breadcrumb to
   diagnose.
3. **Cursor IDE may kill its own child processes** on workspace close.
   `child.unref()` only detaches from the event loop; it doesn't make
   the child immune to `SIGHUP` from a closing parent on some
   platforms (process group inheritance).

### Suggested fix (separate PR)

- Capture spawn output to a file under `~/.deeplake/.autoupdate-out/<timestamp>.log`
  so future failures are diagnosable. The log file is rotated (keep last 10).
- Set `process.env.PATH` explicitly on the spawn so the child can find
  `npm` regardless of whether the parent had it.
- Pass `setsid` (Linux) / detached process group on Linux so SIGHUP from
  Cursor doesn't kill the child.
- Verify against `cursor` snap on at least one fresh install.

### What this PR established for Cursor

- ✅ Cursor's session-start hook calls `autoUpdate(creds, { agent: "cursor" })`
- ✅ The trigger fires (cache-file mtime confirmed in real Cursor session)
- ✅ Direct helper invocation via synthetic stdin produces the expected
  `[autoupdate] agent=cursor entered` log line
- ⚠️ The detached `hivemind update` spawned from inside Cursor IDE
  doesn't complete its npm install (env/sandbox-specific)

## OpenClaw — version-check timed out at 5s during test run

### Symptom

Plugin's `checkForUpdate()` (in `openclaw/src/index.ts`, called once on
gateway start) hit `AbortSignal.timeout(5000)` during the E2E test:

  [plugins] Auto-update check failed: The operation was aborted due to timeout

The expected output `⬆️ Hivemind update available: 0.6.99 → ... Run: hivemind update`
never rendered.

### Confirmation that the change itself is in the bundle

  $ grep -oE "registry.npmjs.org/@deeplake|clawhub.ai" \
      ~/.openclaw-dev/extensions/hivemind/dist/index.js
  registry.npmjs.org/@deeplake

  $ grep -c "openclaw plugins update" ~/.openclaw-dev/extensions/hivemind/dist/index.js
  0   # (after the SKILL.md fix landed in commit 4a57786)

So the centralization at the **source-of-truth** layer (npm registry vs
ClawHub) and the **advice-text** layer (`Run: hivemind update`) is in
place. Live verification of the rendered notice didn't happen this run
because of the timeout.

### Confirmation the registry itself is fast

  $ time curl -sS -o /dev/null https://registry.npmjs.org/@deeplake/hivemind/latest
  real 0m0.167s

So 5s timeout is plenty in the steady state. The first-run timeout was
likely cold TLS + bonjour-watchdog noise running concurrently with
plugin register.

### Suggested fix (separate PR)

- Bump the `AbortSignal.timeout(5000)` to `10000` in
  `openclaw/src/index.ts:checkForUpdate` to absorb cold-init latency.
- Or: add a single retry on `AbortError` with exponential backoff.

### What this PR established for OpenClaw

- ✅ Bundle URL changed from `clawhub.ai` → `registry.npmjs.org/@deeplake/...`
- ✅ Advice text changed from `openclaw plugins update hivemind` →
  `Run: hivemind update` (in the in-prompt notice, the registerCommand
  handler, AND the inlined SKILL.md)
- ⚠️ Live notice render not observed this run (5s timeout); registry
  measured 167ms in steady state, so a small bump fixes it

## Sequencing

These follow-ups are **independent** of PR #97. PR #97 should land as-is —
both items above are pre-existing limitations / cosmetic timeout issues,
not regressions introduced by the centralization. The follow-ups can ship
as separate small PRs after #97 merges.
